Recommendations on Solver Robustness & Testing


As the yearly MaxSAT Evaluation drives MaxSAT solver development forward, we want to recommend the following standardizations to the solvers to make them easier to use. These suggestions are in line with the mathematical logic of the Boolean satisfiability problem.

  • An empty instance should yield a weight "o 0", with an empty model line "v" and the status line "s OPTIMUM FOUND".
  • An empty hard clause should result in an unsatisfiable instance.
  • An empty soft clause should be considered unsatisfied by all solutions and thus incur cost.
  • A solver should accept weight 0 for a soft clause.

We emphasize that all benchmark sets used in the main track evaluation only contain non-empty clauses and at least one soft clause. These suggestions should be taken as recommendations and topics for discussion. Similar topics are currently discussed in the SAT solving community as well where some solvers, like kissat and CaDiCaL, can handle empty instances/clauses, while others cannot.

MaxSAT Fuzzer and Regression Suite

We further want to advertise the use of a regression suite which provides a set of small valid instances known to have caused issues in a previous solver. All solvers submitted to the main tracks need to pass the regression suite. More specific information can be found on the GitHub repository or the rules page.


Finally, we would like to bring your attention to a MaxSAT fuzzing toolsuite for exhaustive testing of MaxSAT solvers available on Github. This toolsuite compares the solution of your solver with any existing solver which complies to the rules of the MaxSAT Evaluation. It compares if the "o-value" is the same as the value provided by your model and if no other solver (e.g. a verified MaxSAT solver) can produce a better solution output with a correct model. Then it minimizes the instance with a delta debugging algorithm to provide simple instances for debugging.
